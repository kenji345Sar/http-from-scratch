# 用語集

`step6.md` や `flask-equivalent.md` で出てくる Python Web まわりの用語を、
「普段の意味」と「このプロジェクトのどこに対応するか」で並べたもの。
読みながら詰まったときの逃げ場として使う。

---

## WSGI（読み: ウィスキー / ダブリュ・エス・ジー・アイ）

- **正式名**: Web Server Gateway Interface
- **普段の意味**: 「Python の Web **サーバ** と Web **アプリ** が話すときのルール」。
- **「Flask は WSGI 互換」と言うとき**:
  Flask が「サーバとアプリのあいだの決まった会話形式」に従って書かれている、ということ。
  その形式に従っていれば、サーバ側は gunicorn でも waitress でも、自前 `server.py`
  でも、アプリを乗せ換え可能。
- **似たもの**:
  Java の Servlet、Ruby の Rack、Node.js の Express の req/res 関係に近い立ち位置。
- **このプロジェクトでの場所**:
  step6 で書いた `server.py` の `build_environ()` と `start_response` の取り決め。
  境界線は [`docs/step6.md`](step6.md) の「server.py に残ったもの / app.py に出ていったもの」の表。

## PEP 3333

- **正式名**: Python Enhancement Proposal 3333
- **普段の意味**: PEP は「Python の公式仕様書フォーマット」。番号が振られていて、
  3333 番目が WSGI の仕様。
- **「PEP 3333 で決まっている」と言うとき**:
  「Python の公式仕様として WSGI のルールはここに書いてある」という参照。
  WSGI ＝ PEP 3333 と読み替えてだいたい合っている。
- **本物**: <https://peps.python.org/pep-3333/>

## Werkzeug（読み: ヴェルクツォイク）

- **意味**: ドイツ語で「道具」。
- **普段の意味**: 「Flask の **下回りライブラリ**」。Flask は Werkzeug の上に
  乗っている薄い枠組みにすぎない。
- **やっていること**:
  - HTTP リクエストをオブジェクトにする (`Request` クラス)
  - レスポンスをオブジェクトにする (`Response` クラス)
  - ルーティング
  - 開発用 dev サーバ (`werkzeug.serving`)
  - つまり **step1〜6 で自前で書いた処理の実運用版**
- **覚え方**: 「Flask の `request.args` `request.form` の実体は Werkzeug の Request」。
- **本物**: <https://github.com/pallets/werkzeug>

## environ 辞書

- **普段の意味**: WSGI で、**サーバがアプリに「これがリクエストですよ」と渡す辞書**。
- **直訳**: environ = environment（環境）。「リクエストが置かれている環境」のイメージ。
- **中身の例**:

  | キー | 値の例 |
  |---|---|
  | `REQUEST_METHOD` | `"GET"` |
  | `PATH_INFO` | `"/hello"` |
  | `QUERY_STRING` | `"name=tsk"` |
  | `CONTENT_TYPE` | `"application/x-www-form-urlencoded"`（あれば） |
  | `HTTP_HOST` | `"127.0.0.1:8080"` |
  | `wsgi.input` | ボディを読むファイルライク |

- **このプロジェクトでの場所**:
  step6 の [`build_environ`](../server.py) で組み立てているのがまさにこれ。

## start_response

- **普段の意味**: WSGI で、**アプリがサーバに「ステータスとヘッダはこれです」と
  伝えるための関数**。
- **呼び方**: `start_response("200 OK", [("Content-Type", "text/plain")])`
- **このプロジェクトでの場所**:
  step6 の `server.py` の `call_app()` の中で定義していて、`app.py` 側から呼ばれる。

## gunicorn（読み: ガニコーン）

- **意味**: Green Unicorn の略。
- **普段の意味**: **本番環境で Python の Web アプリを動かすサーバ側ソフト**。
- **特徴**:
  - Linux / macOS で定番
  - プロセスをいくつも立ち上げて並列にリクエストをさばく（pre-fork 方式）
  - WSGI 互換のアプリならなんでも乗せられる
- **どこで出てくるか**:
  Flask の `app.run()` は「開発用、本番では使うな」と Flask 自身が警告する。
  本番では次のように起動する:

  ```
  gunicorn flask_equivalent:app
  ```

  ※ `flask_equivalent:app` は「`flask_equivalent.py` の中の `app` オブジェクト」の意味。
- **このプロジェクトでの対応**:
  自前の `server.py` が「ミニ gunicorn」の役。

## waitress（読み: ウェイトレス）

- **普段の意味**: gunicorn と同じく **WSGI 用のサーバ側ソフト**。
- **特徴**:
  - 純 Python 製
  - Windows でも動く（gunicorn は基本 UNIX 系のみ）
- **使い方**: `waitress-serve --listen=127.0.0.1:8080 flask_equivalent:app`
- **gunicorn との使い分け**:
  - UNIX 系で本番: gunicorn が定番
  - クロスプラットフォーム / 小さめのデプロイ: waitress

## ハンドラ / view 関数

- **普段の意味**: 「あるリクエストが来たときに呼ばれる関数」。
  名前は文脈によって変わるが、指しているものは同じ。
- **このプロジェクトでの呼び方**:
  - step3〜5: `handle_root` / `handle_hello` など → 「ハンドラ」
  - step6: `view_root` / `view_hello` など → 「view 関数」（Flask の呼び方に寄せた）

## ルーティング

- **普段の意味**: 「リクエストの **method + path** から、呼ぶべきハンドラを選ぶ仕組み」。
- **このプロジェクトでの場所**:
  step3 で初めて登場、`ROUTES` 辞書 + `dispatch()` がこれ。
  Flask だと `@app.route("/")` 1 行で済む。

## 関係図（再掲）

```
あなたが書く側  ┐
               │  Flask アプリ（app = Flask(...)）        ← アプリ層
               │           │
               │      WSGI（PEP 3333）                    ← 「会話ルール」
               │           │
あなたが        │  Werkzeug or gunicorn / waitress         ← サーバ層
書かない側      │           │                                （このプロジェクトでは server.py）
               │     TCP socket / OS
               ┘
```

- **アプリ層** = Flask（`flask_equivalent.py` がこれ）
- **会話ルール** = WSGI（仕様書は PEP 3333）。「アプリ層」と「サーバ層」を独立に作れる理由
- **サーバ層** = 開発時は Werkzeug の dev サーバ、本番は gunicorn / waitress。
  自前 `server.py` も同じ立ち位置
