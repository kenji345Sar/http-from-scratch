# 認証・認可の概念整理

`http-from-scratch` の続編として、認証 (authentication) と認可 (authorization)
を「TCP の上にどう積み上がっているか」という視点で整理する。

このファイルは **概念とランドスケープ** だけを扱う。
実装は後続の `step1.md`〜 でやる。

---

## 1. 認証と認可の違い

| 観点 | 認証 (AuthN) | 認可 (AuthZ) |
|---|---|---|
| 問い | 「あなたは誰か？」 | 「あなたに何を許すか？」 |
| 入力 | ID / パスワード / トークン / 生体 | 認証済みの「誰」+ 操作対象 |
| 出力 | ユーザを特定する (= subject が決まる) | 許可 / 拒否 |
| 失敗時のステータス | `401 Unauthorized` | `403 Forbidden` |
| 例 | 「これは確かに山田さんだ」 | 「山田さんは経理レポートを見ていい」 |

ポイント:

- **認証が成功しても、認可は別途行われる**。ログインできた = 何でもできる、ではない。
- **HTTP のステータスコードは紛らわしい**: `401 Unauthorized` は実は「認証されていない」を意味する。「権限がない」は `403`。

## 2. 認証の主要方式

リクエストごとに「あなたは誰か」をどう示すかの違い。

### 2.1 Basic 認証

```
Authorization: Basic <base64(user:pass)>
```

- 毎リクエストに ID/PW を載せる。
- base64 は暗号ではない。**HTTPS が必須**。
- 社内ツールや API の踏み台でだけ生き残っている。

### 2.2 セッション認証（Cookie + サーバ側ストア）

```
ブラウザ          サーバ
  ─── POST /login (id, pw) ──►
  ◄── Set-Cookie: sid=abc123 ──
  ─── GET /me  Cookie: sid=abc123 ──►
  ◄── (サーバ側で sid → user を引く) ──
```

- 認証情報は **サーバ側** にあり、ブラウザは引換券 (`sid`) だけを持つ。
- 状態をサーバが持つので **ステートフル**。
- 伝統的な Web アプリ（Rails / Laravel / Django のデフォルト）はだいたいこれ。

### 2.3 トークン認証 (JWT など)

```
Authorization: Bearer <JWT>
```

- トークン自体に「誰か」「いつまで有効か」が **署名つきで埋め込まれている**。
- サーバ側にセッション表が要らない (ステートレス)。
- マイクロサービスや SPA / モバイルアプリと相性が良い。
- 弱点: いったん発行したトークンは **取り消しにくい**（だから短命にする / refresh token と組み合わせる）。

### 2.4 OAuth 2.0 / OIDC（Sign in with Google など）

- **自前で ID/PW を預からない** 方式。認証は Google / GitHub / Apple に任せ、結果のトークンだけ受け取る。
- OAuth 2.0 は本来「認可の委譲」の規格、OIDC はそれを認証用に拡張したもの。
- 「Sign in with Google」のボタンの裏で動いているのはこれ。

### 2.5 パスキー / WebAuthn (FIDO2)

- パスワードをやめて、公開鍵暗号で「秘密鍵を持っている本人」を証明する。
- 秘密鍵は端末の TPM / Secure Enclave に閉じる。サーバには **公開鍵だけ** が登録される。
- フィッシング耐性があり、現在の業界推奨。

### 2.6 まとめ

| 方式 | サーバが持つ状態 | 主な用途 | 現状の位置づけ |
|---|---|---|---|
| Basic | なし | 内部 API | レガシー寄り |
| セッション | セッション表 | 一般的な Web アプリ | 今でも王道 |
| JWT | なし（鍵だけ） | SPA / マイクロサービス | API で広く使われる |
| OAuth/OIDC | クライアント登録情報 | 外部ログイン連携 | 事実上の標準 |
| パスキー | 公開鍵 | パスワードレス | 推奨方向 |

## 3. 認可の主要モデル

「誰に何を許すか」をどう表現するか。

### 3.1 RBAC (Role-Based Access Control)

- ユーザに **ロール** (admin / editor / viewer) を持たせ、ロールに権限を紐付ける。
- 一番よく使われる。`@admin_required` のようなデコレータで実装されがち。

### 3.2 ABAC (Attribute-Based Access Control)

- 「部署が経理 かつ 申請者が自部署 のとき許可」のように **属性の式** で判定する。
- 細かい制御ができるが、ルールが増えると爆発する。

### 3.3 ACL (Access Control List)

- 「このファイルは user_42 と user_77 が読める」のように **オブジェクト単位** で許可者リストを持つ。
- ファイルサーバや Google Drive の共有設定が典型。

### 3.4 ポリシーエンジン (OPA / Cedar)

- 認可ロジックをアプリから外に出して、**ポリシー言語** (Rego / Cedar) で書く。
- 大規模・マルチサービスで認可ルールを統一したいときに使う。

## 4. 現状の主要製品マップ

「自作でやるべきところ」と「製品が肩代わりするところ」を把握するための一覧。

### 4.0 全体のレイヤ構造

認証・認可まわりの製品は、3 つのレイヤに分かれている。

| レイヤ | 何をする | 例 |
|---|---|---|
| 3. アプリ側プロトコル | ブラウザ ↔ アプリの認証通信 | OIDC、SAML、Basic |
| 2. IDaaS / 仲介 | プロトコル変換、ポリシー、MFA | Auth0、Okta、Keycloak、ADFS |
| 1. ユーザーディレクトリ | ユーザー・グループの **保管庫** | LDAP、Active Directory |

下から順に「ユーザーがどこに居て」「誰が仲介して」「アプリは何を喋るか」が決まる。

### 4.1 認証 (IDaaS)

| 製品 | 種別 | 特徴 |
|---|---|---|
| Auth0 (Okta) | SaaS | 老舗。エンタープライズ実績豊富 |
| Okta Workforce | SaaS | 社内 SSO の定番 |
| AWS Cognito | SaaS | AWS と統合。安いが UX 寄りでない |
| Firebase Auth | SaaS | モバイル/小規模 SaaS で人気 |
| Clerk | SaaS | 開発者体験重視。SPA/Next.js と相性◎ |
| Supabase Auth | SaaS/OSS | Postgres と一体。OSS でセルフホストも可 |
| Keycloak | OSS | エンタープライズ向けセルフホスト |
| WorkOS | SaaS | SSO/SAML/SCIM の B2B 連携に特化 |

### 4.2 認可

| 製品 / 仕様 | 種別 | 特徴 |
|---|---|---|
| OPA (Open Policy Agent) | OSS | 汎用ポリシーエンジン。Rego で書く |
| AWS Cedar | OSS | AWS 製のポリシー言語 |
| Casbin | OSS ライブラリ | アプリ組み込み型 |
| OAuth 2.0 scope | 仕様 | API ごとの権限を文字列で表現 |
| SpiceDB (Authzed) | OSS/SaaS | Google Zanzibar 系。リレーションベース認可 |

### 4.3 ユーザーディレクトリ (LDAP / AD)

「ユーザー情報の置き場所」を提供するレイヤ。認証フローそのものではなく、
認証フローが裏で参照する **ユーザー DB** に相当する。

| 実装 | 種別 | 立ち位置 |
|---|---|---|
| Active Directory (AD) | Microsoft 製 | 企業 Windows 環境の事実上の標準。中身は LDAP + Kerberos |
| OpenLDAP | OSS | クラシックな LDAP サーバ |
| 389 Directory Server | OSS (Red Hat) | エンタープライズ向け |
| FreeIPA | OSS | LDAP + Kerberos + CA を統合管理 |

補足:

- **LDAP** はディレクトリ (= 木構造のユーザー/グループ DB) にアクセスする TCP プロトコル。
  「ユーザー DB のための SQL」のような位置。
- LDAP には `BIND` 操作があり、「この DN とパスワードで接続できるか？」で
  **認証検証** にも使える (simple bind)。昔の社内アプリは LDAP を直接叩いていた。
- 今は **アプリは LDAP を直接見ない**。アプリは OIDC/SAML を喋り、その裏側で
  Keycloak / ADFS / Auth0 が LDAP/AD をユーザー DB として読みに行く構成が普通。
- つまり LDAP/AD は **バックエンドの真実のソース** として現役、フロントには出てこない。

### 4.4 標準仕様

- **OAuth 2.0 / OIDC**: 認証・認可の通信プロトコル
- **SAML 2.0**: 企業 SSO の古典仕様（今も現役）
- **SCIM**: ユーザ情報を IdP からアプリに同期するための仕様
- **WebAuthn / FIDO2**: パスキーの仕様

## 5.1 ブラウザとサーバの分担 (Cookie / Session / 属性の所在)

セッション認証は「両側で持つもの」と「片側で持つもの」が混在するので混乱しやすい。
1 つの表で整理する。

| 何 | ブラウザ側 | サーバ側 |
|---|---|---|
| Cookie 保管庫 | ◯ (DevTools → Application → Cookies で見える) | × |
| Session 表 | × | ◯ (`SESSIONS` dict / Redis / DB) |
| sid (引換券) | Cookie の値として保存 | `SESSIONS` の key として保存 |
| パスワード本体 | フォーム送信時だけ通る | ハッシュ化して保存 (`USERS`) |
| Cookie の属性 (`HttpOnly` 等) | サーバから渡されたものを守る | **サーバが Set-Cookie で指定する** |

要点:

- **Cookie はブラウザ側、Session はサーバ側**。両者を結ぶのが sid。
- Cookie の **属性は必ずサーバが Set-Cookie で送る**。クライアント (JS / curl) から
  属性を「セット」することはできない。これがセキュリティの肝。
- curl でテストするとき `-c cookies.txt` / `-b cookies.txt` は単に
  「ブラウザの Cookie 保管庫の代わり」をしているだけで、属性の決定権は
  サーバ側にある。

### 流れの図

```
[ブラウザ]                                    [サーバ]
                                              SESSIONS = {}
                                              USERS = {"alice": (salt, hash)}

   POST /login (user=alice, password=...)  ─►
                                              ├─ verify_password OK
                                              ├─ sid = secrets.token_urlsafe(32)
                                              └─ SESSIONS[sid] = "alice"
                                          ◄─  Set-Cookie: sid=abc; HttpOnly; SameSite=Lax
   ├─ Cookie 保管庫に保存
   └─ 属性 (HttpOnly 等) を覚える

   GET /me   Cookie: sid=abc              ─►
                                              ├─ parse_cookies → sid="abc"
                                              └─ SESSIONS["abc"] → "alice"
                                          ◄─  200 OK  "you are alice"
```

### curl とブラウザの違い

| 動作 | curl | ブラウザ |
|---|---|---|
| Cookie の自動保管 | `-c cookies.txt` を付けたときだけ | 自動 |
| Cookie の自動送信 | `-b cookies.txt` を付けたときだけ | 自動 |
| `HttpOnly` の効き目 | (CLI なので JS なし) | JS から `document.cookie` で読めなくなる |
| `SameSite` の効き目 | 無視 | 別ドメインからの POST に Cookie を載せない |
| `Secure` の効き目 | 無視 | HTTPS でしか送らせない |

属性は **ブラウザの "正しい挙動" を制御するためのもの**。
curl では実質効かないが、本番ではブラウザが守ってくれる。

## 6. 用語の整理（混乱しがちなもの）

| 用語 | 意味 |
|---|---|
| Identity Provider (IdP) | 認証を提供する側 (Google, Okta, Keycloak) |
| Service Provider / Relying Party (SP/RP) | IdP に認証を委ねる側 (= 自分のアプリ) |
| SSO | 一度ログインしたら複数サービスにそのまま入れる仕組み |
| MFA | パスワードに加えて第二要素 (SMS / TOTP / パスキー) で本人確認 |
| Access Token | API を叩くためのトークン (短命) |
| Refresh Token | Access Token を再発行するためのトークン (長命) |
| ID Token | 「誰がログインしたか」を表す JWT (OIDC で出てくる) |
| Bearer Token | 「持っている人が使える」型のトークン (= JWT/不透明トークン両方を含む総称) |
