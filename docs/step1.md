# step1: 生 TCP で HTTP リクエストを受け取る

> 前提知識（HTTP / curl / socket の役割分担）は `docs/概念.md` を参照。

## ゴール

- HTTP がフレームワークの中で組み立てられる前に、まず「TCP の上を流れる
  ただのテキスト」であることを確認する。
- Python の `socket` だけで、ポート 8080 を listen し、受け取ったバイト列を
  そのまま標準出力に表示するサーバを書く。
- レスポンスは「最低限の HTTP 形式の固定文字列」を返すだけにする。
  ちゃんとした組み立ては step4 で扱う。

## やること

1. TCP ソケットを開く（`AF_INET`, `SOCK_STREAM`）。
2. `127.0.0.1:8080` に bind して listen する。
3. クライアントが繋いできたら、受け取ったバイト列を画面に出す。
4. 固定の HTTP レスポンスを返してコネクションを閉じる。
5. これをループで繰り返す。

ポイントは「HTTP のパース・ルーティング・レスポンス生成は、まだ一切やらない」
こと。step1 では「TCP に文字列が流れている」ことだけを目で見る。

## 実行手順

サーバ起動：

```
cd /Users/apple/Desktop/Site/http-from-scratch
python3 server.py
```

別ターミナルから curl で叩く：

```
curl -v 'http://127.0.0.1:8080/'
curl -v 'http://127.0.0.1:8080/hello?name=tsk'
curl -v -X POST -d 'a=1&b=2' 'http://127.0.0.1:8080/submit'
```

URL は **必ずシングルクォート `' '` で囲む**。zsh は `?` を「任意の 1 文字に
マッチする glob」、`&` を「バックグラウンド実行」として解釈してしまうため、
クォートしないと `zsh: no matches found` などのエラーで curl が起動しない。
HTTP やサーバの問題ではなく、シェルが URL を食べてしまっている。

ブラウザで `http://127.0.0.1:8080/` を開いてもよい。
（ブラウザは favicon も取りに来るので、リクエストが 2 回飛ぶことに気づける）

サーバを止めるには `Ctrl+C`。

## 確認すること

サーバ側のターミナルに、こういうテキストが見えるはず：

```
GET / HTTP/1.1
Host: 127.0.0.1:8080
User-Agent: curl/8.x.x
Accept: */*

```

ここから読み取りたいこと：

- 1 行目が「リクエスト行」（メソッド + パス + HTTP バージョン）
- 2 行目以降が「ヘッダ」（`名前: 値` 形式）
- 空行のあとに「ボディ」が来る（GET の場合は空）
- 区切りは `\r\n`（CRLF）

POST のときは末尾にボディが付いてくる。`Content-Type` と `Content-Length`
ヘッダが増えていることも見ておく。
