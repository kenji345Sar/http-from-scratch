# 概念: HTTP / curl / socket の役割分担

このプロジェクトを通じて扱う、3 つの登場人物の責務を一度整理しておく。
全 step に共通する前提なので、必要になったら何度でもここに戻ってくる。

## HTTP とは

クライアントとサーバが、TCP の上で取り交わす「テキストの取り決め」。

- クライアントが決まった形式のテキスト（**リクエスト**）を送る
- サーバが決まった形式のテキスト（**レスポンス**）を返す
- 形式は「1 行目 → ヘッダ複数行 → 空行 → ボディ」
- 行の区切りは `\r\n`（CRLF）

リクエストの 1 行目は「リクエスト行」：

```
GET / HTTP/1.1
```

レスポンスの 1 行目は「ステータス行」：

```
HTTP/1.1 200 OK
```

HTTP は動的な何かではなく、**テキストの読み書きの規約**にすぎない。
step1 のサーバ画面で見えていた `GET / HTTP/1.1\r\nHost: ...` が、
まさにそのテキスト本体。

## curl が何をしているか

`curl http://127.0.0.1:8080/` を実行したとき、curl がやっているのは：

1. URL をパース → ホスト `127.0.0.1` / ポート `8080` / パス `/` に分解
2. TCP で `127.0.0.1:8080` に接続
3. `GET / HTTP/1.1\r\nHost: ...\r\n\r\n` というテキストを書き込む
4. サーバから返ってきたテキストを読む
5. ボディ本体を画面に出す（`-v` 付きならヘッダや送信内容も `>` `<` で表示）

つまり curl は「ブラウザの代わりにテキストを送って読む、小さな HTTP クライアント」。
ブラウザがやっていることのうち、HTML を描画する部分を除いた、
**通信の部分だけ**を担当していると言ってよい。

## curl 1 行に何が積み重なっているか

`curl http://127.0.0.1:8080/` というたった 1 行のコマンドの裏では、
**ユーザ空間（curl 本体）と カーネル空間（TCP/IP/loopback）の 2 つの世界**が積み重なって動いている。

上が curl に近い層、下が物理に近い層。

| 層 | 担当 | 何をしているか |
|---|---|---|
| ① curl 本体 | ユーザ空間のアプリ | URL を `host=127.0.0.1 / port=8080 / path=/` に分解。`GET / HTTP/1.1\r\nHost: ...\r\n\r\n` というテキストを組み立てる |
| ② HTTP | テキストの規約 | 「1 行目＋ヘッダ＋空行＋ボディ」「行末は `\r\n`」というフォーマット。プログラムではなく **取り決め** |
| ③ ソケット API | OS が curl に貸す窓口 | `socket()` `connect()` `send()` `recv()`。curl はここを叩いてバイト列を出し入れする |
| ④ TCP | OS カーネル | 3 ウェイハンドシェイク、順序保証、再送。「信頼できるバイトストリーム」を提供 |
| ⑤ IP | OS カーネル | 宛先 `127.0.0.1` にパケットをルーティング |
| ⑥ loopback (lo0) | OS カーネル | `127.0.0.1` は物理 NIC（Wi-Fi / 有線）を使わず、カーネル内で折り返す。**LAN にも外にも出ない** |

押さえておきたいのは 2 点。

- **HTTP は ③〜⑥ の上に乗っているテキストにすぎない**。
  curl がやっているのは「① でテキストを作って ③ から流し込む」だけで、
  TCP より下は全部 OS がやってくれる。
- **`127.0.0.1` は外に出ない**。
  ⑥ の loopback で完結するので、ネットワーク機器もファイアウォールも介在しない。
  だから step1 のサーバが手元の 1 台で完結する。

step1 で書いている `server.py` は、curl と対称な側で同じスタックに乗っている。
curl が ① でテキストを作って ③ に流すのに対し、server.py は ③ からバイト列を受け取って、
そのテキストを「人間の目で」読んでいるのが今の段階。

## curl 本体は具体的にどう組み立てているか

上の表の ① では「curl 本体が HTTP テキストを組み立てる」と書いたが、
これは抽象的な話ではなく、**curl の C ソースに最初から書き込まれている処理**にすぎない。
curl は C で書かれたオープンソースの CLI で、ソースは <https://github.com/curl/curl> にある。

### 「GET」が決まる場所

curl は CLI オプションを見て、メソッドをこのルールで決める。

| 指定 | メソッド |
|---|---|
| 何も指定なし | **GET** |
| `-d` / `--data` | POST |
| `-T` / `--upload-file` | PUT |
| `-X METHOD` | 指定どおり |
| `-I` | HEAD |

`curl http://127.0.0.1:8080/hello?name=tsk` は何も指定がないので、GET と決まる。

### リクエスト文字列が組み立てられるまで

curl の中の処理段階をざっくり見るとこうなる。

| 段階 | curl ソース上の場所（目安） | やっていること |
|---|---|---|
| CLI パース | `src/tool_getparam.c` | `-X` `-d` `-H` などを解釈し、メソッド・ヘッダ・ボディを決める |
| URL パース | `lib/url.c` | `http://127.0.0.1:8080/hello?name=tsk` を host/port/path/query に分解 |
| リクエスト組み立て | `lib/http.c` の `Curl_http()` 周辺 | `"GET /hello?name=tsk HTTP/1.1\r\n"` + `Host:` + `User-Agent:` + `Accept:` を順に文字列追加 |
| 送信 | `lib/sendf.c` → ソケット層 | 組み立てたバイト列を `send()` で TCP ソケットに書き込む |

つまり server.py 側の画面に出てくる

```
GET /hello?name=tsk HTTP/1.1
Host: 127.0.0.1:8080
User-Agent: curl/7.68.0
Accept: */*
```

は、curl の C ソース内で `printf` / `sprintf` 相当の文字列操作によって組み立てられた結果が、
そのまま TCP 経由で server.py に届いたもの。

### ここから何が言えるか

- 「GET」「Host」「User-Agent」「Accept」という文字列は、**curl の C ソースにテンプレートとして直書きされている**。動的にどこかから降ってくるわけではない。
- step1 の `server.py` が `recv()` で読んでいるのは、その文字列の到着物そのもの。
- HTTP の意味（メソッド名・ヘッダ名）は、curl 側にも「文字列のテンプレート」としてしか入っていない。**curl も socket と同じく「テキストを組み立てて流すだけ」**で、HTTP の意味そのものを誰か別の人が解釈しているわけではない。

ここまで踏み込むと、step2 以降で server.py に積む「HTTP パーサ」とは、
**curl が組み立てたテキストを逆向きに分解する処理**だと見えてくる。

## server.py の socket が何をしているか

socket は **TCP の口を開けて、バイト列を読み書きするだけ**。
HTTP のルールは何も知らない。

| 呼び出し | 役割 |
|---------|------|
| `socket(AF_INET, SOCK_STREAM)` | TCP ソケットを作る |
| `bind((HOST, PORT))` | そのポートを自分のものにする |
| `listen()` | 接続待ち状態にする |
| `accept()` | クライアントが来るまで停止 → 来たら専用コネクションを返す |
| `recv()` | クライアントが送ってきたバイト列を読む |
| `sendall()` | 任意のバイト列を返す |

step1 のキモは、「socket は HTTP を理解していないのに、curl が送ってきた
**たまたま HTTP 形式のテキスト** がそのまま見える」という点。
HTTP の意味（メソッド、パス、ヘッダ、ステータスコード）を解釈しているのは
現時点ではどちらでもなく、人間の目だけ。

## Python の socket は OS への薄いラッパー

上の表に出した `socket()` `bind()` `listen()` `accept()` `recv()` `sendall()` は、
**Python が独自に何かを実装しているのではない**。すべて OS のシステムコールに
ほぼ 1 対 1 で対応している。

| Python | OS のシステムコール（Linux / macOS） |
|---|---|
| `socket.socket(AF_INET, SOCK_STREAM)` | `socket(2)` |
| `s.bind((host, port))` | `bind(2)` |
| `s.listen(5)` | `listen(2)` |
| `s.accept()` | `accept(2)` |
| `s.recv(n)` | `recv(2)` / `read(2)` |
| `s.sendall(bytes)` | `send(2)` / `write(2)` |

呼ぶと、こういう流れで OS まで降りていく:

```
あなたのコード（server.py）
   │ s.accept()
   ▼
Python ランタイム（CPython の C 実装）
   │ accept(fd, ...) ← C のシステムコール呼び出し
   ▼
OS カーネル
   │ TCP の状態管理（接続テーブル、送受信バッファ、3-way handshake）
   ▼
NIC / loopback
```

**TCP のプロトコル状態（誰と繋がっているか、バッファに何が溜まっているか）は
OS カーネルが持っている**。Python の socket オブジェクトは、カーネルに
お願いするための窓口（ファイルディスクリプタという整数 ID）を Python から
触れるようにしているだけ。

つまり「Python が TCP を理解している」のではなく、**「OS が TCP を理解していて、
Python はそれを呼んでいる」**。
他の言語（Go の `net.Listen` / Java の `ServerSocket` / Node の `net.createServer`）
も、結局やっているのは同じシステムコールへの問い合わせ。

## TCP の本体はカーネルにある（socket では見えない）

`server.py` を読み返すと気づくが、**TCP に関するコードはどこにも書かれていない**。
唯一「TCP」を頼んでいるのはこの 1 行だけ:

```python
socket.socket(socket.AF_INET, socket.SOCK_STREAM)
                                  ─────────────
                                  ↑
                       これが「TCP を使う」の宣言
```

`SOCK_STREAM` という定数 1 個が **「TCP のソケットをくれ」を OS に頼む合図**。
これを `SOCK_DGRAM` にすれば UDP になる。socket API は **どのプロトコルか
選ぶ場所だけ提供していて、実装は全部 OS カーネル側**。

### `recv()` の裏でカーネルがやっている TCP の仕事

```python
raw = conn.recv(4096)
```

この 1 行で受け取るのは **「順番に並んだ・欠損のない・エラーチェック済みの
バイト列」**。ここまでに OS が裏でやっていることは多い。

| TCP の仕事 | あなたが見えるか | やっている主体 |
|---|---|---|
| 3-way handshake (SYN / SYN-ACK / ACK) | 見えない。`accept()` で返ってきた時点で完了済み | OS カーネル |
| パケット分割 (MSS で切る) | 見えない | OS カーネル + NIC |
| シーケンス番号で並べ直す | 見えない | OS カーネル |
| 重複パケットを捨てる | 見えない | OS カーネル |
| ACK を返す | 見えない | OS カーネル |
| ロスト時の再送 | 見えない | OS カーネル |
| フロー制御 (window size) | 見えない | OS カーネル |
| 輻輳制御 (CUBIC / BBR 等) | 見えない | OS カーネル |
| チェックサム検証 | 見えない | OS カーネル |
| TIME_WAIT 状態の維持 | `SO_REUSEADDR` を設定するときに少しだけ意識する | OS カーネル |
| 4-way close (FIN / ACK) | 見えない。`with conn:` の出口で `close()` するだけ | OS カーネル |

`recv()` が返す `bytes` は、地球の向こうから断片で届いたかもしれない数十・
数百の TCP パケットを、**カーネルが再構築してくれた結果**にすぎない。

### 全部の層を縦に見る

```
あなたのコード (server.py)
   │   "recv で 4KB ほしい"
   ▼
Python の socket モジュール（CPython の C 実装）
   │   recv() システムコール
   ▼
OS カーネルの TCP/IP スタック   ← ★ ここに TCP の全部
   │   Linux なら net/ipv4/tcp_*.c という C のコードが何万行も
   ▼
NIC ドライバ
   ▼
ネットワーク
```

★ の「TCP/IP スタック」は、Linux カーネルの C ソースで `tcp_input.c` /
`tcp_output.c` などに数万行のコードとして実装されている。
**この数万行が、`recv()` 1 行の裏で動いている**。

### TCP を覗き見るには別の道具が要る

socket からは TCP の機構そのものは見えないので、見たければ別の道具を使う。

| 道具 | 見えるもの |
|---|---|
| `tcpdump` / Wireshark | 実際の TCP パケット（SYN/ACK/seq/データ） |
| `netstat` / `ss` | 今どの IP・ポートで TCP コネクションが張られているか、状態（ESTABLISHED, TIME_WAIT 等） |
| `/proc/net/tcp` (Linux) | 上記の生データ |
| Python の `socket` レベル | 結果としてのバイト列だけ。TCP の機構は見えない |

### どの言語でも同じ

```go
// Go
ln, _ := net.Listen("tcp", ":8080")
```

```java
// Java
ServerSocket server = new ServerSocket(8080);
```

```javascript
// Node.js
net.createServer(...)
```

```csharp
// C#
new TcpListener(IPAddress.Loopback, 8080).Start();
```

どの言語も結局やっているのは **「カーネルに『TCP のソケットを用意してくれ』
と頼んで、あとはバイトを読み書きする」**だけ。
TCP プロトコル本体は **カーネルに 1 つしかなくて、それを全言語が共有している**。

### まとめ

- `server.py` に TCP の処理は **1 文字も書かれていない**
- 唯一「TCP」を選んでいるのは `SOCK_STREAM` 定数だけ
- TCP の数万行の実装は **OS カーネルにあり、socket API はその窓口**
- 「socket が TCP を隠している」というより、**「TCP は最初から OS の中にしか
  なく、socket はそこへのアクセス口」** という関係
- これは Python 固有ではなく、どの言語でも同じ

## このプロジェクトでは TCP の両側を同じカーネルがやっている

ここまでで「TCP は OS カーネルの中にある」と書いた。ではこのプロジェクトでは
**どこの OS の TCP が動いているのか**を、はっきりさせておく。

### 構図

このプロジェクトでは、curl も `server.py` も **同じ MacBook 上で走っている**。
だから、TCP の処理は **両方とも macOS カーネル（Darwin の TCP/IP スタック）が
やっている**。

```
  curl プロセス                          server.py プロセス
  ─────                                ─────────
  (あなたの Mac)                          (あなたの Mac)
       │                                     │
       │  socket() で「TCP ソケットくれ」      │  socket() で「TCP ソケットくれ」
       ▼                                     ▼
  ┌────────────────────────────────────────────────────────┐
  │              macOS カーネル                            │
  │              ─────────────                             │
  │   ┌──────────┐                       ┌──────────┐     │
  │   │ curl 用の│ ─── SYN ───────────► │ server用 │     │
  │   │TCPソケット│ ◄── SYN-ACK ──────── │TCPソケット│     │
  │   │          │ ─── ACK ───────────► │          │     │
  │   │          │ ─── HTTP データ ────► │          │     │
  │   └──────────┘                       └──────────┘     │
  │        ↑                                   ↑          │
  │        └─── loopback (lo0) で繋がる ───────┘          │
  └────────────────────────────────────────────────────────┘
                       ▲
                       └ 物理 NIC は使わない（外に出ない）
```

「TCP を喋っている 2 人」（client 側 / server 側）が、たまたま
**同じ家（macOS カーネル）の中で会話している** 状態。

### 普段の Web ではここが変わる

ブラウザで google.com を開くとき:

```
あなたの Mac                            Google のサーバ
─────                                ─────
 macOS カーネル      ──TCP/IP──►       Linux カーネル
 (Darwin の実装)     ◄──TCP/IP──        (Linux の実装)

  → 物理 NIC → Wi-Fi/有線 → ルータ → 海底ケーブル → データセンタ
```

- **両側のカーネルは別物**（あなたのは Darwin、Google のは Linux）
- ただし **TCP は世界共通のプロトコル（RFC 9293）** なので、別実装でも会話できる
- 実際のパケットが物理ネットワークを飛ぶ

### よくある勘違いの整理

- **「同じマシン内だから handshake は省略してる？」** → していない。loopback
  でも 3-way handshake もシーケンス番号も再送ロジックも全部発生する。
  ただ物理層に降りないだけで、**「TCP-over-メモリ」** のような形で本物の
  TCP が走っている
- **「同じカーネルだから socket は共有されてる？」** → されていない。curl 用と
  server.py 用は別オブジェクト。**お互い「別人」だと思って話している**
- **「loopback だと速い？」** → 速い。物理層を経由しないので Wi-Fi 経由より
  ずっと速い。ただし TCP の意味的な動作は完全に同じ

### 関係表

| 観点 | このプロジェクト | 本物の Web |
|---|---|---|
| client 側 TCP を誰がやる | macOS カーネル | あなたの OS のカーネル |
| server 側 TCP を誰がやる | **同じ** macOS カーネル | サーバ側 OS（多くは Linux）のカーネル |
| 物理ネットワークを通るか | 通らない（loopback で完結） | 通る |
| 速さ | とても速い | 物理距離次第 |
| TCP の意味的動作 | 全部発生する（handshake / ACK / 再送 等） | 同じく全部発生する |
| 異常系（パケットロス、遅延） | ほぼ起きない | 普通に起きる |

つまり、このプロジェクトで触れているのは **本物の TCP**。
ただ参加者 2 人がたまたま 1 つのカーネルの中に居るので、外に出ないだけ。
プロトコルとしての TCP の振る舞いは、本番と同じものを学べている。

## ルータ・物理ネットワーク機器はアプリから見えない

ここまでで「TCP の本体はカーネルにある」「このプロジェクトでは両側を同じ
macOS カーネルがやっている」と書いた。
ではその下の **ルータ・Wi-Fi・ISP・ファイアウォール** はどうなっているか。

### このプロジェクト: 1 台も登場しない

127.0.0.1 は loopback で完結するので、**物理ネットワーク機器は 1 つも関与しない**。

- 自宅 Wi-Fi ルータ → 無関係
- 職場のルータ・スイッチ → 無関係
- ISP のルータ → 無関係
- 海底ケーブル → 無関係
- ファイアウォール / NAT → 無関係
- ロードバランサ → 無関係

**ネットワーク機器が 1 つも要らない状態**で TCP/HTTP を練習している、と理解
していて差し支えない。

### 本物の Web: 大量に介在しているが、アプリには見えない

ブラウザで google.com を開くとき、実際にはこれくらい挟まっている:

```
あなたの Mac                            Google のサーバ
─────                                ─────
ブラウザ                                server プロセス
   │                                          │
TCP/IP (kernel)                          TCP/IP (kernel)
   │                                          │
NIC                                          NIC
   │                                          │
家の Wi-Fi ルータ → ISP のルータ → バックボーンルータ → DC のロードバランサ → サーバ
              ↑                    ↑                    ↑                    ↑
              全部「アプリからは見えない」インフラ
```

- アプリは「`google.com:443` に繋いで」と OS に頼むだけ
- 「どのルータを経由するか」「どの経路で運ぶか」は **アプリではなく
  OS と経路上の各ルータが決めている**
- アプリが体感するのは結果だけ:「繋がった / タイムアウト / refused」

### 「ルータ」という単語の 2 つの意味を分けておく

| 「ルータ」の意味 | 実体 | アプリから見えるか |
|---|---|---|
| **物理のルータ装置**（家の Wi-Fi ルータ、業務用 Cisco / Juniper など） | 別マシン。BGP / OSPF / 静的ルートで動く | **見えない** |
| **OS の内部のルーティング決定** | カーネルの routing table（どの NIC・どの gateway に流すか） | **アプリからは見えないが、OS の中に存在する** |

OS 内のルーティング決定は `netstat -rn` や `route -n get default` で覗ける:

```
$ netstat -rn
Destination        Gateway            Flags     ...
default            192.168.1.1        UGScg     ...   ← 外向きはここに送る
127                127.0.0.1          UCS       ...   ← 127.x.x.x は loopback へ
...
```

「`127.0.0.1` 宛は loopback へ、それ以外は default gateway へ」という規則が
カーネルに書かれていて、アプリの代わりに OS がこの判断をしている。

### アプリにネットワーク機器が「見える」瞬間（数少ない例外）

通常は完全に透過だが、稀に顔を出す:

| シナリオ | アプリへの影響 |
|---|---|
| ファイアウォール / NAT で接続拒否 | `connection refused` / `timeout` |
| 経路 MTU 不一致、Path MTU Discovery 失敗 | 大きいパケットが届かなくなる |
| ルータ故障 | `network unreachable` |
| 特定インタフェースから出したい | `SO_BINDTODEVICE`（Linux）など、ほぼ使わない |

つまり **「失敗の症状」としてしか見えない**。成功している間、アプリは何も知らずに済む。

### 早見表

| 層 | このプロジェクト | 本物の Web |
|---|---|---|
| HTTP | アプリ（`server.py`） | アプリ（Flask 等） |
| TCP | macOS カーネル | 両端の OS カーネル |
| IP・ルーティング決定 | macOS カーネル（lo0 行きと判定） | 両端の OS + 経路上のルータ群 |
| 物理ルータ装置 | **登場しない** | 何台も経由（家 / ISP / バックボーン / DC） |
| Wi-Fi / Ethernet / 光ファイバ | **使わない** | NIC・ケーブル・無線・ファイバ |
| ファイアウォール / NAT | 関係なし | あり（多くは透過、失敗時のみ顔を出す） |
| ロードバランサ | 関係なし | あり（ALB / nginx 等、これも透過） |

### 結びとして

- 「**ルータの設定の制御**」は **アプリ層から完全に切り離されている領域**
- ネットワーク管理者 / インフラエンジニアの仕事（AWS なら VPC / セキュリティグループ、
  オンプレなら Cisco IOS の設定 等）
- HTTP / Flask / WSGI のレイヤを学ぶうえでは、一度も触らなくてよい
- このプロジェクトでは loopback のおかげで、そもそも物理ネットワーク機器
  が 1 つも登場しない状態で練習できている

## step1〜6 でずっと socket を直接使っている

socket そのものの呼び方は、どの step でも `server.py` の中で **同じ**。
step を進めても socket 部分は変わっていなくて、変えたのは「socket と socket の
あいだに何を挟むか」だけ。

| step | socket の使い方 | 周辺で増えたもの |
|---|---|---|
| step1 | recv → print → 固定 sendall | （socket だけ） |
| step2 | recv → **parse** → 固定 sendall | parse_request |
| step3 | recv → parse → **dispatch** → sendall | ROUTES + dispatch |
| step4 | recv → parse → dispatch → **build_response** → sendall | text / html / redirect |
| step5 | recv → parse → dispatch → build_response → sendall | クエリ / フォーム / 静的 |
| step6 | recv → parse → **environ** → **call_app** → sendall | WSGI 境界 |

つまり、このプロジェクトの全 step を通して、**socket は常に表に出ている**。

## Flask だと socket はどこに行ったか

Flask のコードを書くとき `import socket` は出てこない。
だが socket は **誰かが必ず使っている**。場所が違うだけ。

| Flask の動かし方 | socket を実際に呼んでいるもの |
|---|---|
| 開発: `flask run` / `app.run()` | **Werkzeug の dev サーバ** |
| 本番: `gunicorn flask_app:app` | **gunicorn** |
| 本番: `waitress-serve ...` | **waitress** |
| 本番: nginx + uwsgi | **uwsgi** |

層で見るとこう:

```
あなたが書く側  ┐
               │  Flask アプリ              ← socket は触らない
               │           │                  （このプロジェクトの app.py に相当）
               │      WSGI 経由
               │           │
書かない側     │  Werkzeug / gunicorn       ← ここで socket を呼ぶ
               │                              （このプロジェクトの server.py に相当）
               │     ↓ socket
               │     OS カーネル              ← TCP の本体
               ┘
```

- **socket を触る役そのものは消えていない**。Flask の世界では、それが
  Werkzeug / gunicorn という別レイヤに **分業されている**だけ
- このプロジェクトでは、その「下の層（socket を呼ぶ層）」も自分で書いている
  ので、socket が表に出ている
- 「Flask が socket を隠蔽している」というより、
  **「Flask アプリの開発者は、socket を呼ぶ役を別人に任せている」** と読むと
  正確

## 三者の関係

```
  curl                              server.py
  ────                              ─────────
  URL をパース                      socket() で TCP の口を開ける
       │                                │
       │  TCP 接続                      │
       ├───────────────────────────────►│ accept()
       │                                │
       │  HTTP リクエスト（テキスト）    │
       ├───────────────────────────────►│ recv() で読む
       │                                │
       │  HTTP レスポンス（テキスト）    │ sendall() で書く
       │◄───────────────────────────────┤
       │                                │
       │  コネクション close            │
       └───────────×────────────────────┘
```

curl と socket のあいだを流れているのは「ただのバイト列」で、
その中身がたまたま HTTP の形式に従っているだけ。

step2 以降で、サーバ側に「HTTP の意味」を少しずつ実装していく。
具体的には：

- step2: 受け取ったテキストを「リクエスト行・ヘッダ辞書・ボディ」に分解する
- step3: パスとメソッドから関数を選ぶ（ルーティング）
- step4: ステータス行とヘッダとボディを自分で組み立てる（レスポンス生成）

ここまで来ると、socket は相変わらず「バイト列の読み書き」しかしないが、
そのバイト列に意味を与えるレイヤが、サーバ側に積み上がっていく。
