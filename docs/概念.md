# 概念: HTTP / curl / socket の役割分担

このプロジェクトを通じて扱う、3 つの登場人物の責務を一度整理しておく。
全 step に共通する前提なので、必要になったら何度でもここに戻ってくる。

## HTTP とは

クライアントとサーバが、TCP の上で取り交わす「テキストの取り決め」。

- クライアントが決まった形式のテキスト（**リクエスト**）を送る
- サーバが決まった形式のテキスト（**レスポンス**）を返す
- 形式は「1 行目 → ヘッダ複数行 → 空行 → ボディ」
- 行の区切りは `\r\n`（CRLF）

リクエストの 1 行目は「リクエスト行」：

```
GET / HTTP/1.1
```

レスポンスの 1 行目は「ステータス行」：

```
HTTP/1.1 200 OK
```

HTTP は動的な何かではなく、**テキストの読み書きの規約**にすぎない。
step1 のサーバ画面で見えていた `GET / HTTP/1.1\r\nHost: ...` が、
まさにそのテキスト本体。

## curl が何をしているか

`curl http://127.0.0.1:8080/` を実行したとき、curl がやっているのは：

1. URL をパース → ホスト `127.0.0.1` / ポート `8080` / パス `/` に分解
2. TCP で `127.0.0.1:8080` に接続
3. `GET / HTTP/1.1\r\nHost: ...\r\n\r\n` というテキストを書き込む
4. サーバから返ってきたテキストを読む
5. ボディ本体を画面に出す（`-v` 付きならヘッダや送信内容も `>` `<` で表示）

つまり curl は「ブラウザの代わりにテキストを送って読む、小さな HTTP クライアント」。
ブラウザがやっていることのうち、HTML を描画する部分を除いた、
**通信の部分だけ**を担当していると言ってよい。

## curl 1 行に何が積み重なっているか

`curl http://127.0.0.1:8080/` というたった 1 行のコマンドの裏では、
**ユーザ空間（curl 本体）と カーネル空間（TCP/IP/loopback）の 2 つの世界**が積み重なって動いている。

上が curl に近い層、下が物理に近い層。

| 層 | 担当 | 何をしているか |
|---|---|---|
| ① curl 本体 | ユーザ空間のアプリ | URL を `host=127.0.0.1 / port=8080 / path=/` に分解。`GET / HTTP/1.1\r\nHost: ...\r\n\r\n` というテキストを組み立てる |
| ② HTTP | テキストの規約 | 「1 行目＋ヘッダ＋空行＋ボディ」「行末は `\r\n`」というフォーマット。プログラムではなく **取り決め** |
| ③ ソケット API | OS が curl に貸す窓口 | `socket()` `connect()` `send()` `recv()`。curl はここを叩いてバイト列を出し入れする |
| ④ TCP | OS カーネル | 3 ウェイハンドシェイク、順序保証、再送。「信頼できるバイトストリーム」を提供 |
| ⑤ IP | OS カーネル | 宛先 `127.0.0.1` にパケットをルーティング |
| ⑥ loopback (lo0) | OS カーネル | `127.0.0.1` は物理 NIC（Wi-Fi / 有線）を使わず、カーネル内で折り返す。**LAN にも外にも出ない** |

押さえておきたいのは 2 点。

- **HTTP は ③〜⑥ の上に乗っているテキストにすぎない**。
  curl がやっているのは「① でテキストを作って ③ から流し込む」だけで、
  TCP より下は全部 OS がやってくれる。
- **`127.0.0.1` は外に出ない**。
  ⑥ の loopback で完結するので、ネットワーク機器もファイアウォールも介在しない。
  だから step1 のサーバが手元の 1 台で完結する。

step1 で書いている `server.py` は、curl と対称な側で同じスタックに乗っている。
curl が ① でテキストを作って ③ に流すのに対し、server.py は ③ からバイト列を受け取って、
そのテキストを「人間の目で」読んでいるのが今の段階。

## curl 本体は具体的にどう組み立てているか

上の表の ① では「curl 本体が HTTP テキストを組み立てる」と書いたが、
これは抽象的な話ではなく、**curl の C ソースに最初から書き込まれている処理**にすぎない。
curl は C で書かれたオープンソースの CLI で、ソースは <https://github.com/curl/curl> にある。

### 「GET」が決まる場所

curl は CLI オプションを見て、メソッドをこのルールで決める。

| 指定 | メソッド |
|---|---|
| 何も指定なし | **GET** |
| `-d` / `--data` | POST |
| `-T` / `--upload-file` | PUT |
| `-X METHOD` | 指定どおり |
| `-I` | HEAD |

`curl http://127.0.0.1:8080/hello?name=tsk` は何も指定がないので、GET と決まる。

### リクエスト文字列が組み立てられるまで

curl の中の処理段階をざっくり見るとこうなる。

| 段階 | curl ソース上の場所（目安） | やっていること |
|---|---|---|
| CLI パース | `src/tool_getparam.c` | `-X` `-d` `-H` などを解釈し、メソッド・ヘッダ・ボディを決める |
| URL パース | `lib/url.c` | `http://127.0.0.1:8080/hello?name=tsk` を host/port/path/query に分解 |
| リクエスト組み立て | `lib/http.c` の `Curl_http()` 周辺 | `"GET /hello?name=tsk HTTP/1.1\r\n"` + `Host:` + `User-Agent:` + `Accept:` を順に文字列追加 |
| 送信 | `lib/sendf.c` → ソケット層 | 組み立てたバイト列を `send()` で TCP ソケットに書き込む |

つまり server.py 側の画面に出てくる

```
GET /hello?name=tsk HTTP/1.1
Host: 127.0.0.1:8080
User-Agent: curl/7.68.0
Accept: */*
```

は、curl の C ソース内で `printf` / `sprintf` 相当の文字列操作によって組み立てられた結果が、
そのまま TCP 経由で server.py に届いたもの。

### ここから何が言えるか

- 「GET」「Host」「User-Agent」「Accept」という文字列は、**curl の C ソースにテンプレートとして直書きされている**。動的にどこかから降ってくるわけではない。
- step1 の `server.py` が `recv()` で読んでいるのは、その文字列の到着物そのもの。
- HTTP の意味（メソッド名・ヘッダ名）は、curl 側にも「文字列のテンプレート」としてしか入っていない。**curl も socket と同じく「テキストを組み立てて流すだけ」**で、HTTP の意味そのものを誰か別の人が解釈しているわけではない。

ここまで踏み込むと、step2 以降で server.py に積む「HTTP パーサ」とは、
**curl が組み立てたテキストを逆向きに分解する処理**だと見えてくる。

## server.py の socket が何をしているか

socket は **TCP の口を開けて、バイト列を読み書きするだけ**。
HTTP のルールは何も知らない。

| 呼び出し | 役割 |
|---------|------|
| `socket(AF_INET, SOCK_STREAM)` | TCP ソケットを作る |
| `bind((HOST, PORT))` | そのポートを自分のものにする |
| `listen()` | 接続待ち状態にする |
| `accept()` | クライアントが来るまで停止 → 来たら専用コネクションを返す |
| `recv()` | クライアントが送ってきたバイト列を読む |
| `sendall()` | 任意のバイト列を返す |

step1 のキモは、「socket は HTTP を理解していないのに、curl が送ってきた
**たまたま HTTP 形式のテキスト** がそのまま見える」という点。
HTTP の意味（メソッド、パス、ヘッダ、ステータスコード）を解釈しているのは
現時点ではどちらでもなく、人間の目だけ。

## 三者の関係

```
  curl                              server.py
  ────                              ─────────
  URL をパース                      socket() で TCP の口を開ける
       │                                │
       │  TCP 接続                      │
       ├───────────────────────────────►│ accept()
       │                                │
       │  HTTP リクエスト（テキスト）    │
       ├───────────────────────────────►│ recv() で読む
       │                                │
       │  HTTP レスポンス（テキスト）    │ sendall() で書く
       │◄───────────────────────────────┤
       │                                │
       │  コネクション close            │
       └───────────×────────────────────┘
```

curl と socket のあいだを流れているのは「ただのバイト列」で、
その中身がたまたま HTTP の形式に従っているだけ。

step2 以降で、サーバ側に「HTTP の意味」を少しずつ実装していく。
具体的には：

- step2: 受け取ったテキストを「リクエスト行・ヘッダ辞書・ボディ」に分解する
- step3: パスとメソッドから関数を選ぶ（ルーティング）
- step4: ステータス行とヘッダとボディを自分で組み立てる（レスポンス生成）

ここまで来ると、socket は相変わらず「バイト列の読み書き」しかしないが、
そのバイト列に意味を与えるレイヤが、サーバ側に積み上がっていく。
