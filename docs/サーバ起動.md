# サーバを起動するということ

> このドキュメントは、「`php artisan serve` や `flask run` や `dotnet run` は
> 何度も打ったことがある。でも、そのコマンドが裏で何を始めているかは、あまり
> 意識せずに使っている」というユーザに向けて書いている。

## 普段のあなたがやっていること

開発で何かを試すとき、フレームワークごとにこういうコマンドを打っているはず。

| あなたが打つコマンド | フレームワーク |
|---|---|
| `php artisan serve` | Laravel |
| `flask run`（または `python xxx.py`） | Flask |
| `python manage.py runserver` | Django |
| `dotnet run` | ASP.NET Core |
| `npm run dev` | Node.js / Express / Next.js |
| `rails s` | Rails |
| `./gradlew bootRun` | Spring Boot |
| `python3 server.py` | このプロジェクト |

それぞれ別のコマンドだが、**裏で起きていることは全部同じ構造**。

## それらのコマンドが裏で始めていること

どのコマンドも、結局これだけのことをやっている。

1. プロセスを 1 つ立ち上げる
2. 指定の TCP ポート（8000 / 8080 / 3000 / 5000 など）を OS から借りる（`bind`）
3. 「リクエストが来るまで止まって待つ」ループに入る（`accept` ループ）
4. 来たリクエストに応じてコードを呼び、レスポンスを返す
5. `Ctrl+C` で殺されるまで、それを延々続ける

このプロジェクトの [`server.py`](../server.py) の `serve_forever()` が、まさに
この 5 つを文字通り書いたもの。`php artisan serve` でも `dotnet run` でも、
中身がリッチになっているだけで、骨格は同じ。

## なぜ「自分で起動」する必要があるか

Web は **クライアント** と **サーバ** の対話で成り立っている。

```
クライアント側                       サーバ側
─────                             ─────
ブラウザ / curl   ──HTTP─────►   常駐プロセス
                  ◄──HTTP────    （リクエストを待っている）
```

- **クライアント側**（あなたが URL を入力するブラウザ）はリクエストを送るだけ
- **サーバ側**（リクエストを受ける側）は、**誰かが起動して、止めない限り動き続けている**プロセス

普段ネットサーフィンしているときは、サーバ側を **Google / YouTube / 会社のクラウド** が動かしてくれている。だから「URL を入れるだけ」で済む。

開発しているときは、自分が書いたコードを動かす **サーバ側プロセスは誰も用意してくれない** ので、自分で `php artisan serve` などを打って起動する。これが「コマンドを打つ理由」。

## 「常駐しているのは何か」は言語によって違う

ここがよく混乱するポイント。`php artisan serve` のときと `dotnet run` のときでは、
**常駐しているもの**が違う。

### A. アプリのプロセスがそのまま常駐するモデル

- **対象**: Python / Node.js / Go / Ruby / Java / **.NET Core (ASP.NET Core)** / Rust
- **常駐しているもの**: あなたのアプリのプロセス（中に HTTP サーバを内蔵）
- **例**:
  - Flask アプリ = Python プロセス（Werkzeug の dev サーバ内蔵）
  - ASP.NET Core = `dotnet` プロセス（Kestrel という HTTP サーバ内蔵）
  - Spring Boot = JVM プロセス（Tomcat 内蔵）
- **本番**: 同じプロセスを `systemd` / `Docker` / `Kubernetes` が「死なないように」管理する。
  前段に nginx を置くこともあるが、アプリ自体は常駐し続けている。
- **このプロジェクトの `server.py` はこのモデル**。

### B. アプリは「呼ばれる側」、Web サーバが常駐するモデル

- **対象**: **PHP（伝統的）** / **ASP.NET Framework（古い .NET）** / 古典 CGI
- **常駐しているもの**: Web サーバ（Apache / nginx / IIS / php-fpm）
- **アプリ本体**: リクエストごとに「呼ばれて」「処理して」「終わる」
- **例**:
  - Laravel 本番 = nginx が常駐、リクエストごとに php-fpm の worker が Laravel のコードを呼ぶ
  - ASP.NET Framework = IIS が Windows サービスとして常駐、リクエストごとに `.dll` を呼ぶ
- **このモデルだと**: 「自分でサーバを起動した記憶がない」と感じやすい。Web サーバ（IIS / Apache）は OS と一緒に起動しているし、デプロイは「ファイルを置く」だけだったりするから。
- **例外**: Laravel に Octane（Swoole / RoadRunner）を導入すると、Laravel 自体が常駐するようになり、A モデルに変わる。

### 表で対比

| Framework | 開発時の起動 | 本番で常駐するもの | モデル |
|---|---|---|---|
| 自前 (このプロジェクト) | `python3 server.py` | Python プロセス | A |
| Flask | `flask run` | Python プロセス（gunicorn） | A |
| Django | `python manage.py runserver` | Python プロセス（gunicorn / uwsgi） | A |
| Laravel | `php artisan serve` | nginx + php-fpm | **B** |
| ASP.NET Core | `dotnet run` | `dotnet` プロセス（Kestrel） | A |
| ASP.NET Framework | （なし）  | IIS | **B** |
| Express (Node) | `npm run dev` | Node プロセス（pm2） | A |
| Rails | `rails s` | Ruby プロセス（puma） | A |
| Spring Boot | `./gradlew bootRun` | JVM プロセス | A |

## 「`php artisan serve`」と「ブラウザで開く」の関係

ここまでをまとめると、`php artisan serve` を打ったとき、こうなっている:

```
あなたの terminal                   あなたの terminal もしくはブラウザ
─────                              ─────
php artisan serve                  curl http://127.0.0.1:8000/
↑ サーバ側プロセスを                   ↑ クライアント側として叩く
   localhost:8000 に常駐させる
↓ ログが流れてくる                     ↓ レスポンスが返ってくる
[2026-05-20 ...] GET /              <html>...</html>
```

つまり:

- **「コマンドを打つ」 = サーバ側プロセスをローカルに立ち上げる**
- **「ブラウザで開く / curl で叩く」 = クライアント側として、ローカルのサーバにアクセスする**

ふだん Web を使うときは「サーバ側がどこかで誰かに動かされている」のがデフォルトなので、
**「サーバ側を自分で起動する」という工程が見えなくなっている**。開発時は自分でやる必要があるので、コマンドとして表に出てくる。

## terminal に出てくるあのログは何

サーバ側プロセスが受け付けたリクエストの記録。

| 環境 | ログの行き先 |
|---|---|
| 今のあなたのローカル | サーバを起動した terminal にそのまま print |
| 共有開発サーバ | サーバの `/var/log/...` などのファイル |
| 本番 (AWS) | CloudWatch Logs / Datadog |
| 本番 (GCP) | Cloud Logging |
| Vercel / Heroku | 管理画面の Logs タブ |

普段「Web 開発しているときログで確認している」と感じているのは正しくて、それは **`php artisan serve` を打った terminal に流れているのと同じ性質のもの**。本番ではログ集約サービスに流れているだけ。

## まとめ

| 問い | 答え |
|---|---|
| なぜ自分でコマンドを打って起動するの？ | サーバ側プロセスは「自動では立ち上がらない」から。本番ではクラウドや IIS が立ち上げているのを、ローカルでは自分でやる |
| `php artisan serve` と `python3 server.py` は何が違うの？ | コマンドが違うだけ。中で起きていることは同じ（TCP ポートを開けて `accept` ループに入る） |
| なぜ Laravel だけ「常駐しているのは nginx」だったり違うの？ | 言語の伝統的な実行モデルが違う（A モデル vs B モデル）。アプリが常駐するか、Web サーバが常駐するかの違い |
| terminal に出てくるあの行は何？ | サーバ側プロセスのログ。本番でも同じ内容がログ集約サービスに流れている |

このプロジェクトでは、A モデルの最小形（`server.py` が常駐 + `accept` ループ）を
自分の手で書いている。`php artisan serve` を打ったときに裏で動き出すものの、
最も骨が見える形のもの、と思って読み返すと、対応が取りやすいはず。
