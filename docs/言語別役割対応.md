# Web フレームワークの「役割」を言語別に見比べる

`gunicorn` / `WSGI` / `Werkzeug` のような Python の言葉は、いきなり並ぶと
とっつきにくい。が、これらは **「Web の仕事を分担する役割の名前」** で、
PHP / Java / C# でも **同じ役割が別の名前で必ず存在している**。

このページは、その「役割 ↔ 各言語での名前」の対応表。

## まず役割の地図

Web リクエストが処理されるまでに、こういう役割の階段がある。

```
クライアント (browser / curl)
    │
    ▼
[① リバースプロキシ]  ← 前段で TLS 終端、ロードバランス、静的配信
    │
    ▼
[② アプリサーバ・コンテナ]  ← プロセスを常駐させ、リクエストを受ける
    │
    ▼
[③ サーバ ⇄ アプリの会話ルール]  ← 両者がデータを受け渡す形式
    │
    ▼
[④ HTTP 抽象ライブラリ]  ← Request / Response オブジェクト
    │
    ▼
[⑤ Web フレームワーク]  ← @app.route、ORM、テンプレート 等
    │
    ▼
あなたが書く view 関数
```

step1〜6 で自前で書いたものは、上の **② ③ ④ ⑤ をぜんぶ小さく自作した**もの。
普段のフレームワーク開発では、これらが全部「既製品」に置き換わる。

## 役割 ↔ 各言語での名前（横断表）

| 役割 | **Python** | **PHP** | **Java** | **C#** |
|---|---|---|---|---|
| ⑤ Web フレームワーク | **Flask** / Django / FastAPI | **Laravel** / Symfony / CodeIgniter | **Spring Boot** / Spring MVC | **ASP.NET Core** (Razor Pages / MVC) |
| ④ HTTP 抽象ライブラリ（Request/Response の実体） | **Werkzeug**（Flask 内蔵） | Laravel **`Illuminate\Http\Request`**（下回りは Symfony HttpFoundation） | **Servlet API** (`HttpServletRequest`) | `Microsoft.AspNetCore.Http`（`HttpContext`） |
| ③ サーバ ⇄ アプリの会話ルール | **WSGI**（PEP 3333）/ ASGI | **PSR-15**（モダン PHP）/ 伝統的 mod_php 直結 | **Servlet API** / Jakarta EE | ASP.NET Core **middleware pipeline** |
| ② アプリサーバ・アプリコンテナ | **gunicorn** / uwsgi / waitress | **php-fpm**（FastCGI） | **Tomcat** / Jetty / Undertow | **Kestrel**（.NET 内蔵） |
| ① リバースプロキシ | nginx / Apache | nginx / Apache | nginx / Apache | nginx / **IIS** |
| 開発時の dev サーバ | `flask run`（Werkzeug dev） | `php artisan serve`（PHP `-S`） | `bootRun`（組み込み Tomcat） | `dotnet run`（Kestrel） |
| ルーティングの書き方 | `@app.route("/")` | `Route::get('/', ...)` | `@RequestMapping("/")` | `[Route("/")]` / `MapGet("/", ...)` |
| クエリ取得 | `request.args.get("k")` | `$request->input('k')` | `request.getParameter("k")` | `Request.Query["k"]` |
| フォーム取得 | `request.form.get("k")` | `$request->input('k')` | `request.getParameter("k")` | `Request.Form["k"]` |

## このプロジェクトの自前実装が、どの役割か

| 役割 | 普段のフレームワーク | このプロジェクトのファイル |
|---|---|---|
| ⑤ Web フレームワーク | Flask など | `app.py` の view 関数 + ROUTES |
| ④ HTTP 抽象 | Werkzeug の `Request` / `Response` | `parse_request()` + `build_response()`（`server.py`） |
| ③ 会話ルール | WSGI（規約） | `build_environ()` + `start_response`（`server.py`） |
| ② アプリサーバ | gunicorn | `server.py` の `serve_forever()` |
| ① リバースプロキシ | nginx | **なし**（学習用に省略） |

つまり [`server.py`](../server.py) は、各言語の世界で言うと:

- Python 世界: **gunicorn のミニ版**
- Java 世界: **Tomcat のミニ版**
- C# 世界: **Kestrel のミニ版**
- PHP 世界: **php-fpm のミニ版**

に相当する。

[`app.py`](../app.py) は **Flask アプリのミニ版** で、Java 世界の Spring Boot
アプリ、C# 世界の ASP.NET Core アプリと同じ位置に居る。

## step1〜6 が、表の役割 ①〜⑤ のどこを進めたか

このプロジェクトは「サーバを起動する側」と「アプリを書く側」の全部を、
小さく自分の手で書いていく構成。各 step が、上の役割表のどの位置を
進めたものかを並べる。

| step | 書いたもの | 担当した役割 | Flask だと | Laravel だと | Java だと | C# だと |
|---|---|---|---|---|---|---|
| step1 | `socket.accept` / `recv` / `sendall` + 固定 sendall | **②** アプリサーバの土台（接続を受ける部分） | Werkzeug dev server / gunicorn の accept ループ | php-fpm の worker accept | Tomcat の Connector | Kestrel の listen |
| step2 | `parse_request` | **④** HTTP 抽象（リクエスト側） | Werkzeug の `Request` 構築 | `Illuminate\Http\Request` 構築 | `HttpServletRequest` 構築 | `HttpContext.Request` 構築 |
| step3 | `ROUTES` + `dispatch` | **⑤** ルーティング | `@app.route("/")` | `Route::get('/', ...)` | `@RequestMapping("/")` | `MapGet("/", ...)` / `[Route("/")]` |
| step4 | `build_response` + `text` / `html` / `redirect` | **④** HTTP 抽象（レスポンス側） | Werkzeug の `Response` / `make_response` | `response()` / `redirect()` | `HttpServletResponse` | `HttpResponse` / `Results.Ok()` |
| step5 | `parse_qs_simple` / `handle_static` | **④** クエリ・フォーム + **⑤** 静的配信 | `request.args` / `request.form` / `static_folder` | `$request->input()` / `public/` 配下 | `request.getParameter` / `WebMvcConfigurer.addResourceHandlers` | `Request.Query` / `UseStaticFiles` |
| step6 | `build_environ` + `start_response`、`app.py` 分離 | **③** サーバ⇄アプリの会話ルール（WSGI 境界） | WSGI そのもの（PEP 3333） | PSR-15 相当（伝統 PHP は mod_php / php-fpm 直結） | Servlet API の `service(req, resp)` 契約 | ASP.NET Core middleware の `HttpContext` 引き渡し |

### どこから何が見えるようになっていったか

- **step1**: いちばん下層。「TCP の口を開けて接続を受ける」だけ。
  これがフレームワークが立ち上がるときに最初にやっていること。
- **step2〜4**: 受け取ったバイト列 → リクエストオブジェクト → レスポンスオブジェクト
  の「中身」。Werkzeug / HttpFoundation / Servlet API / HttpContext が普段
  やっている処理を自前で書いた。
- **step5**: クエリ・フォームのパース、静的配信。**フレームワークの
  「日常機能」** が初めて自分の手で動く段階。
- **step6**: 「**アプリ ⇄ サーバ**」の境界線そのもの (WSGI = PEP 3333) を
  自前で実装した。これ以降は、`app.py` を Flask に差し替えても、
  `server.py` を gunicorn に差し替えても動く形になる。

つまり step1〜6 は **下層から上層に向かって、フレームワークの内側を
一段ずつ可視化する**構成になっている。

各 step の実装の細かい話は [`stepN.md`](.) を、Flask 版でやるとどうなるかは
[`flask-equivalent.md`](flask-equivalent.md) を参照。

## 逆引き: 「○○ で言う Y は、Python では何？」

### PHP の人向け

| PHP で出てくる名前 | Python での相当 |
|---|---|
| Laravel | Flask / Django |
| Laravel の `Illuminate\Http\Request`（`$request`） | Werkzeug の `Request` |
| **php-fpm** | **gunicorn** |
| `php artisan serve` | `flask run` |
| `Route::get('/', ...)` | `@app.route('/')` |
| `$request->input('k')` | `request.args.get('k')` |
| Composer の autoloader | Python の `import`（明示的に書く） |
| middleware（Laravel） | WSGI middleware |

### Java の人向け

| Java で出てくる名前 | Python での相当 |
|---|---|
| Spring Boot | Flask / Django |
| `HttpServletRequest`（Servlet API） | Werkzeug の `Request` |
| **Tomcat** / Jetty | **gunicorn** |
| Servlet container | WSGI server |
| `./gradlew bootRun` | `flask run` |
| `@RequestMapping("/")` | `@app.route("/")` |
| `request.getParameter("k")` | `request.args.get("k")` |
| Filter / Interceptor | WSGI middleware |

### C# の人向け

| C# で出てくる名前 | Python での相当 |
|---|---|
| ASP.NET Core | Flask / Django |
| `HttpContext` の Request / Response | Werkzeug の `Request` / `Response` |
| **Kestrel** | **gunicorn**（または Werkzeug dev サーバ） |
| `dotnet run` | `flask run` |
| `MapGet("/", ...)` / `[Route("/")]` | `@app.route("/")` |
| `Request.Query["k"]` | `request.args.get("k")` |
| middleware（`UseXxx`） | WSGI middleware |
| **IIS** | **nginx** |
| (旧)ASP.NET Framework + IIS | （該当少。強いて言えば mod_python だが今は廃れ） |

## 「立ち位置」を素早く掴むためのワンライナー

- **WSGI** ≒ Java の **Servlet API**。「アプリとサーバの取り決め」。
- **Werkzeug** ≒ Java の **Servlet 実装本体**。Flask が借りているライブラリ。
- **gunicorn** ≒ Java の **Tomcat**、C# の **Kestrel**、PHP の **php-fpm**。
  アプリを常駐させる「アプリサーバ」。
- **Flask** ≒ Java の **Spring Boot**、C# の **ASP.NET Core**、PHP の **Laravel**。
  ルーティングや view を書く層。
- **nginx / Apache / IIS** はどの言語でも前段に居る同じ役割。
